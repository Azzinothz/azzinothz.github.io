浅谈微信小程序与Web应用和传统移动APP的联系与对比

微信小程序

微信小程序是一种全新的连接用户与服务的方式，它可以在微信内被便捷地获取和传播，同时具有出色的使用体验。

这是来自微信小程序官方文档，官方对于小程序的一句话概括介绍。作为开发者，如果进行过小程序开发和Web开发，那可能你会感觉两者有十分相似的开发体验。

但是呢，小程序并不是Web应用（用微信看网页），两者实质上有一定的区别。

微信小程序与Web应用

首先我们明确一下runtime environment的概念：简单来说就是运行时环境，一段用来跑我们写的代码的代码。

浏览器（中的JavaScript引擎）或是Node.js（中的JavaScript引擎）都是runtime的一种，决定了我们的JavaScript代码能干什么，例如window.alert('Hello World')只有浏览器能理解，而require('fs').readFile('/');也只有Node知道是什么意思。

微信小程序也是实现了JavaScript的runtime environment的一种。

浏览器作为一个runtime的另一个特点是有UI绘制和用户交互行为的捕获能力（HTML、CSS、JavaScript）。而小程序也有这一特点，但其实它并不能识别HTML和CSS（可能是为了进行后续优化，但我觉得目前主要的目的还是为了约束开发者，这在后文与APP对比时会提到），它自己搞出了WXML（对应HTML）和WXSS（对应CSS），似乎后来我好像还看到了WXScript…

客户端开发的核心是通过 runtime 来调度和控制 runtime 之下的平台能力，浏览器这个 runtime 下面的平台是操作系统（Windows、macOS、iOS、Android、*nix 等），而小程序这个 runtime 下面的平台是微信，这是二者的本质区别。

微信小程序与传统APP

一般而言，一个我们手机上常用的APP大小通常都在20MB以上，加上现在国内安卓环境相当恶劣，许多APP的大小都较初期发生了很大程度的膨胀。

而微信小程序，有个2MB的明确的代码包大小限制（以前是1MB）。这虽然很有利于精简程序内容，使小程序的内存占用极少，给用户更好的体验（因为往往用户会选择使用这个APP只有一个或很少的需求），但这也使小程序相比传统APP能做到的规模和功能完整度上失去了竞争力，孰好孰坏还是取决于用户的需求。

除此之外，传统APP在开发存在着许多麻烦的地方，首当其冲的是手机的适配。企业在开发APP使，为了兼顾所有用户，不得不至少开发两版（Android和iOS，WP应该已经算是凉透了），更不用说还需要根据手机的CPU，甚至是屏幕开发更多的版本。同时，传统APP的开发周期也是相对较长的，一个功能完善的APP一般没个一两个月是不太可能上架商用的，我曾经略微涉足过Android APP的前端开发，可以体会到还是比较繁琐的（也有可能是因为我没有系统地学过移动开发…）。

在这一方面，微信小程序的优势就凸显而出。一般小程序的开发周期只要数周，而且产出的产品可以运行在所有主流设备上。相比传统APP需要在发布前通过各个平台的审核（尤其是安卓），而且通常每个平台的审核还需要提交不同的材料，小程序只需要提交到微信公众平台进行审核，就可以很快地上架。

最后还是要讲一讲关于消息推送的问题。我们都知道由于一些原因，Google在大陆需要科学上网才能访问，这可能很多人觉得对自己影响不大，但对于安卓来说其实是相当重大的一个问题。大家都觉得iOS很快很顺畅，其中很大一部分原因是其推送机制，简单来说，各APP的推送是先告诉苹果的服务器，然后由苹果再通过我们手机后台运行的一个IM服务程序推送给我们APP想告诉我们的信息。其实，安卓在国外也是这样的一个类似的推送机制，主要是通过FCM（Firebase Cloud Messaging）进行推送的（以前是GCM），但是由于Google是个不存在的公司，我们没法用上FCM，那么各大厂商就自己想办法推送安卓消息了，这就导致了APP们干脆就一直挂在后台自己给手机推送消息，这引发的问题就是安卓手机给人普遍很卡的感觉，还有就是电池续航的问题等。

回归正题，小程序在推送上也有自己的规定，它只能回复模版消息，不允许主动给用户发送广告。不过因为限制了推送模版，小程序在定制性上有所欠缺，不能像传统APP那样根据自己的需求推送消息。

上面虽然用很大篇幅说了小程序有多好多方便，不过其规模限制和推送限制等问题决定了它目前更适合作为一种即用即走的工具使用。我觉得小程序有一定的发展前景，尤其是在线下领域可能会大放异彩。
